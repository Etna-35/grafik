# Etna · no-money-no-honey

Внутренний корпоративный кабинет ресторана-бара Etna (Вологда). Раздел `grafik` (график смен) уже существует как одностраничное приложение `index.html`. Развивается в кабинет с PIN-входом, ролями и разделами: график, закрытие смены, задачи, выплаты, ЛК сотрудника.

Пользователей максимум 15 всего, одновременно 2–5. Это НЕ высоконагруженный проект — не усложнять.

- `docs/PROJECT.md` — продукт, модель данных, текущий функционал, бэклог, план фаз. Читать перед любой задачей по структуре, данным или миграции.
- `DECISIONS.md` — журнал принятых решений (что и почему решено). Прочитать первым, чтобы не переделывать готовое и не пересматривать закрытые вопросы.
- `docs/STATE.md` — снимок текущего состояния (что реально построено и где). Прочитать вместе с DECISIONS, чтобы знать актуальную точку.

## Как работать

- Сначала уточняющие вопросы и письменный план, потом код. Не бросаться сразу писать реализацию.
- Двигаться мелкими шагами, показывать структуру файлов до реализации.
- TDD не обязателен. Достаточно, чтобы собранное запускалось и работало.
- Ничего не закладывать «на вырост» под нагрузку: масштаб 2–5 одновременных пользователей.
- Задачи приходят уже сформулированными (см. `docs/codex-task-template.md`). Дешёвая модель — для мелких чётких задач; неоднозначное/архитектурное/отладка — на думающую модель.

## Архитектурные инварианты

- Рендер и расчёты не обращаются к `localStorage` напрямую — только через слой доступа к данным (после фазы 0). Это точка, куда позже подключается API.
- При любой правке схемы `state` — обновлять `migrate()` и `schemaVersion`, иначе ломается импорт старых бэкапов.
- Значение смены типизировать (часы vs рубли — разные сущности, не одно перегруженное поле).
- Backend владеет схемой БД и миграциями. NocoDB — только просмотр и лёгкие правки, схему через него не менять.
- Роль-скоуп проверяется в backend, не в NocoDB: сотрудник видит только свою зарплату/смены/задачи. Фронт ходит в backend, а не напрямую в NocoDB API.
- Postgres наружу не публикуется (только docker-сеть). Backend и NocoDB — только через Caddy по HTTPS, с авторизацией.
- PostgreSQL хостим сами в контейнере. Managed-базу (в т.ч. Beget DBaaS) НЕ использовать.

## Визуальный стиль

- **Тёмная тема «RestForm» (единый бренд Etna).** Палитра и шрифты взяты 1:1 из соседнего проекта-формы `Etna-35/RestForm` — это источник правды по визуальной идентичности. Фон `--paper:#0e0e0e`, поверхности `--card:#181818`/`--card-2:#222222`, текст `--ink:#f0ead8` (тёплый кремовый), приглушённый `--ink-soft:#8a8070`, границы `--line:#2e2e2e`.
- Акцент — **антик-золото** `--brand:#c8a96e`, светлое `--brand-bright:#e2c98a`. Текст НА золотой заливке всегда тёмный — `--brand-ink:#0e0e0e` (не белый!). Акцент-текст на тёмном фоне — `--brand-bright`.
- Шрифты: дисплейный **Unbounded** (`--font-display`: логотип, заголовки, золотые uppercase-метки `.section-title/.sec`, клавиши PIN, крупные суммы, золотые кнопки) + текстовый **Golos Text** (`--font`: тело, поля). Playfair/Inter/Manrope убраны.
- Все цвета — только через CSS-переменные в `:root`. Захардкоженных hex в правилах не оставлять (ломает тёмную тему). Успех/опасность — `--ok-bg/--ok-line/--green` (`#5ab87a`) и `--danger/--danger-bg/--danger-line` (`#e05a4a`); warn/`--gold` `#e0a84a`.
- Роли: `--cook` (красный) / `--bar` (тил) / `--waiter` (терракота) / `--dish` (слейт) — подобраны различимыми и не конфликтующими с золотом бренда.
- Откат на старую светлую палитру — `apps/web/styles.light.css.bak`.
- `index.html` в корне — легаси standalone-«График» (светлый, Manrope+Playfair), боевой раздел живёт в `apps/web`. Не путать.
- Метки минималистичные и монохромные, не emoji. Mobile-first.

## Секреты и безопасность

- Секреты только через `.env` (в `.gitignore`), заполняет пользователь руками. В код и git ничего чувствительного не писать.
- Не выполнять действий с реальными кредами, деплоем на прод-сервер, доступами и платежами — это делает пользователь. Агент готовит код и инструкции.
