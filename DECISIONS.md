# Журнал решений · Etna · no-money-no-honey

Зачем: фиксировать принятые решения, чтобы любой агент (Codex/Claude), входя в проект, с одного взгляда видел, что и почему решено, и не переделывал готовое и не пересматривал закрытые вопросы.

Как вести: новые решения дописывать СВЕРХУ, с датой. Формат: `решение — что затрагивает`. Если решение отменяет прежнее — так и писать («отменяет от …»).

## 2026-06-04 — обучение / база знаний и квиз / аттестация

- **Обучение / База знаний** — отдельный раздел в ЛК, доступ по индивидуальным правам сотрудника. На первом этапе основной сценарий — официанты, но модель прав не должна быть жестко привязана только к роли.
- **Контент обучения** хранится в Postgres и редактируется без кода через NocoDB/админку: 7-10 глав, `Маршрут стажера` по дням, вложения к главам (методички, артефакты, ТТК, PDF/изображения/ссылки).
- **Стартовая база знаний официанта** импортируется из Google Doc пользователя `Библия` напрямую в Postgres. Полный текст методички НЕ коммитить в публичный git; после импорта источником правды становится живая БД.
- **Квиз / Аттестация** — отдельный поддомен `quiz.no-money-no-honey.ru`, но тот же backend, те же PIN-аккаунты и та же БД. Не делать отдельную авторизацию и отдельную базу.
- **Результаты квиза** идут в ЛК сотрудника как личный мотиватор и в дашборд руководителя: кто сколько прошел, активность, лучшие результаты, что осталось.
- **Правила по умолчанию:** проходной порог темы 80%, общая аттестация = пройдено/всего, пересдачи разрешены, в зачет идет лучший результат, лидерборд выключен по умолчанию из-за приватности.

## 2026-06-03 — уточнения из вида руководителя

- **Доступ к разделам — по индивидуальным правам.** «Закрытие смены» и «Заявка» доступны не всем; руководитель выдаёт право на раздел в админке (закрытие — отдельным сотрудникам; заявка — обычно повара/бармены). Не чисто по роли.
- **Заявка: категории каталога фильтруются по роли/правам.** Бармену не показывать категории вроде «Мясо». Привязка категорий к ролям/правам — часть модели.
- **Кальянщик — ОТМЕНЯЕТ прежнее «один кальянщик на смену».** Руководитель назначает кальянщиков; в форме закрытия смены появляется СПИСОК назначенных; по каждому отмечается, сколько/кому выплачено; сумма идёт в доходы этого сотрудника в его ЛК.
- **Задачи** выдаются из админ-аккаунта; функционал только доделан, ещё не протестирован.

## 2026-06-03 — уточнения по текущему состоянию ЛК

- **Оценки** — цветные точки на календаре: руководитель отмечает, как сотрудник себя показал (нерегулярно). Уже реализовано, role-scoped (сотрудник видит только свои). Это отдельная сущность, НЕ задачи.
- **Выплаты** ведутся руками на календаре (даты выплат) и role-scoped показываются в ЛК и календаре сотрудника. Отдельный раздел «Выплаты» НЕ нужен — это календарь + вид в ЛК. Заработок кальянщика встраивается в этот же вид.
- **Задачи** — отдельный недоделанный блок. Заведён бриф `docs/codex-brief-tasks.md`. Задачи ≠ оценки.
- Каждый сотрудник видит общий календарь, но свои выплаты и оценки — только свои.

## 2026-06-03 — стартовая фиксация

### Инфраструктура / хостинг
- Стек: Postgres + небольшой backend + NocoDB на одном VPS, Docker Compose. Managed-базу НЕ используем (Beget DBaaS и Яндекс Managed отклонены как избыточные для масштаба).
- Площадка: VPS Beget в РФ (СПб), конфигурация 2 CPU / 4 GB / 40 GB, Ubuntu 24.04. Под масштаб 2–5 одновременных пользователей не закладывать ничего тяжелее нужного.
- Надёжность: самостоятельный Postgres + ежедневный `pg_dump` + копия за пределы сервера; еженедельный бэкап Beget — как страховочная сетка.
- NocoDB — только админка руководителя, НЕ публичный интерфейс для сотрудников.
- Backend владеет схемой БД и миграциями. Роль-скоуп проверяется в backend, не в NocoDB.

### Закрытие смены
- Переносим только форму, её логику и запись данных. Усложнения старой системы НЕ переносим: фудкост 25%, перенос/накопленный остаток закупа, кэш-флоу/прогноз, регулярные платежи, сверка счёта, бот расходов, аналитический дашборд (дашборд можно собрать позже отдельно).
- Логика (проверена на реальных строках): Безнал = Терм.1 + Терм.2 + Я.Еда; Выручка = Безнал + Наличные + Переводы; Остаток (расч.) = Откр. факт + Наличные − (Мойка + Выпл. кальяны + Доп. расходы) − Инкассация; Такси НЕ вычитается из кассы; Разница = Остаток факт − Остаток расч.
- Перенос: Откр. (расч.) = Остаток (факт) предыдущей смены.
- Подсветка расхождения кассы при |Разница| > 500; закрытие не блокируется.
- План выручки — из модуля «График» (единый источник). Если график отдаёт диапазон — для % выполнения берём ВЕРХНЮЮ границу.
- В смене один кальянщик — поле «Кальяны» одно.
- Запись: одна запись на дату в `shift_closings`. Telegram-отчёт при закрытии сохраняем (бот за `.env`).

### Доп. обязанности (кальянщик)
- Доп. обязанность назначается поверх основной роли, со своей оплатой. Первая — «Кальянщик» (ставка ₽/шт), задаётся на сотруднике там же, где роль/часы/ставка.
- Количество кальянов из закрытия смены × ставка → в выплаты этого сотрудника и в статистику его ЛК.
- Таблицы `employee_duties`, `duty_earnings` добавляются в `data-model.md` (делает Codex).

### Заявка (продукты + хозтовары)
- Заявка = список «что нужно купить». НЕ списывает бюджет и НЕ является закупом; фудкост считается отдельно (при закрытии смены). Этот смысл сохранить.
- Два набора: продукты и хозтовары (Тип на категории/товаре).
- Каталог сидируется из `docs/etna-tovarnaya-matrica.xlsx` (листы «Товары» и «Категории»; 202 реальные позиции, 12 категорий).
- Статусы: новая → принята → закуплена → отклонена. Флаг «срочно» — на заявку, не на позицию.
- Остаток/цену сотруднику в MVP не показывать. Разрешить свободную позицию «другое».
- Роль-скоуп: сотрудник видит свои заявки, руководитель — все, меняет статусы.

### Процесс работы
- Источник правды — репозиторий. `AGENTS.md` читается агентом автоматически; решения действительны, только если закоммичены.
- Разделение: Claude — архитектура, брифы, мокапы, промпты; пользователь коммитит; Codex реализует.
- Claude правит только планировочные доки/брифы/мокапы, НЕ трогает сгенерированный Codex код и `data-model.md` напрямую (изменения модели оформляются как инструкции в брифе).
- «Сначала план, потом код». TDD не обязателен.
- Не перезаписывать вслепую доки, доработанные Codex (`PROJECT.md`, `data-model.md`, `service-architecture.md`, `implementation-roadmap.md`, `deployment-beget.md`).
