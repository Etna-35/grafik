# Касса — планировщик денежного потока (owner-only)

Статус: спецификация (2026-06-12). Раздел **только для собственника** (role = `owner`).
Идея и обсуждение: см. чат + макеты. Эта фича — forward-слой поверх существующих Финансов
(`finance.ts`: `predictRevenue`, нормы затрат, ФОТ из графика), но считает **живые деньги на счёте
(cash-flow)**, а не P&L.

---

## 1. Зачем

Помочь владельцу **гарантированно накопить на расчётном счёте к нужной дате** под крупные обязательные
платежи (ЖКХ, аренда, налоги, долги), не залезая в операционные деньги (продукты/напитки/хоз/ФОТ).
Система прогнозирует по средней выручке: хватит ли к дате, сколько откладывать в день, и что делать,
если не хватает (перенести / урезать / сплитовать). Плюс ручные корректировки на форс-мажоры.

## 2. Ключевая концепция — приоритетный водопад + конверты

Выручка каждого дня распределяется **по приоритету**, сверху вниз:

1. **ФОТ — гарантированно, с верха** (из графика, `schedule_shifts.pay_amount`). Святое, всегда обеспечивается первым.
2. **Закуп (бар+кухня)** — ставка `purchase_pct` (**≤30%**, владелец меняет периодически). Идёт в **конверт**.
3. **Хоз** — ставка `household_pct` (**≤5%**, меняется). Идёт в свой **конверт**.
4. **Резерв** — под «точки платежей» (ЖКХ/аренда/налог/долг).
5. **Свободно** — что осталось. Настоящая прибыль/манёвр.

### Конверты бюджета (закуп, хоз) — главная фишка

- **Накопление:** каждый закрытый день конверт += `фактическая_выручка_дня × ставка`.
- **Перенос (rollover):** конверт **не обнуляется** — копится изо дня в день, **ждёт дня заявки**.
- **Списание:** ручной ввод факта закупки (продукты+алко → конверт закупа; хозтовары → конверт хоз).
- **Показатель:** «сколько можно потратить» = текущий баланс конверта (накоплено − потрачено).

Инвариант: **точка платежа никогда не залезает в ФОТ и конверты.** Резервируется только то, что осталось
после ФОТ + закуп% + хоз%. Это делает подсказки безопасными.

### Факт vs прогноз

- **Конверты и «сколько могу потратить»** — от **фактической** выручки (закрытия смен).
- **Обеспеченность будущих платежей** — от **прогнозной** выручки (`predictRevenue`), по остатку после
  ФОТ + закуп% + хоз%.

## 3. Сущности (данные)

- **Остаток на счёте** (`treasury_balance`) — якорь прогноза. Снимок: `balance`, `as_of`, `note`.
  Последний снимок = текущая точка отсчёта. Хранится история (для сверки прогноза с фактом).
- **Точки платежей** (`treasury_payments`) — `title`, `amount`, `due_date`, `category`, `priority`,
  `splittable`, `recurring` (none/monthly/...), `status` (planned/paid/moved/split), `parent_id` (для сплита),
  `paid_at`.
- **Факт трат** (`treasury_spend`) — ручной ввод по 4 категориям: `kind`
  (`food` продукты кухня+бар / `household` хозка / `personal` личные с р/с / `other` форс-мажор),
  `amount`, `spent_on`, `note`. `food`/`household` списывают одноимённые конверты; `personal`/`other`
  уменьшают свободное/резерв.
- **Настройки** (`treasury_settings`) — `purchase_pct` (≤30), `household_pct` (≤5), `safety_buffer` (подушка,
  фикс сумма). ФОТ берётся из графика, не из %. Ставки владелец меняет периодически (история ставок — для
  пересчёта накоплений конвертов задним числом по периодам действия ставки).
- **Категории платежей** — фикс список: `ЖКХ · Аренда · Налог · Долг · Закуп(крупный) · Прочее` (+ свободное
  название). Категория — для цвета/группировки.
- Прогноз выручки **переиспользуем из finance** (`predictRevenue`).

## 4. Алгоритм (cash-flow прогноз)

1. **Старт:** `B0` = последний снимок остатка. Горизонт = сегодня … `max(due_date)` + буфер.
2. **Конверты (от факта):** по закрытым дням `конверт_закуп = Σ(факт_выручка × purchase_pct) − Σ(spend.food)`;
   `конверт_хоз = Σ(факт_выручка × household_pct) − Σ(spend.household)`. Это и есть «сколько можно потратить».
   Ставки берём по периодам их действия (история в `treasury_settings`).
3. **Дневной свободный поток (для прогноза платежей):**
   `FCF[d] = прогноз_выручки[d] − ФОТ[d] − прогноз_выручки[d] × (purchase_pct + household_pct)`,
   где `ФОТ[d]` — из графика. Только этот остаток идёт в резерв/свободно.
4. **Покрытие платежей:** платежи по возрастанию `due_date`. Накопленный `FCF` распределяется по очереди;
   `coverage = min(amount, доступно_к_дате)`. Платёж **обеспечен**, если `coverage ≥ amount`,
   иначе `shortfall = amount − coverage`.
5. **Сколько откладывать в день:** `остаток_нужно / дней_до_срока` (ограничено дневным `FCF`).
6. **Свободно сегодня:** `FCF[сегодня] − Σ(рекомендованные отчисления на сегодня)`.
7. **Подсказки при нехватке:** ① ближайшая дата, где платёж обеспечен (перенос);
   ② платежи ниже приоритетом, которые можно подвинуть до срока (на сумму нехватки);
   ③ сплит: сколько реально к сроку + остаток позже.
8. **Профицит:** если резерв перекрывает все платежи + подушку — излишек = «реально свободно»,
   предложение направить на долги.
9. **Факт трат (`treasury_spend`)** списывает конверты/свободное; `other` (форс-мажор) пересчитывает прогноз.

## 5. API (всё gated `role = owner`)

- `GET  /api/treasury` — текущий остаток, платежи[], корректировки[], расчёт (покрытие по платежам,
  отчисление/день, флаги обеспеченности, разбивка по трём корзинам).
- `POST /api/treasury/balance` — задать снимок остатка.
- `POST|PUT|DELETE /api/treasury/payments[/:id]` — точки платежей.
- `POST /api/treasury/payments/:id/pay` · `.../split` — оплатить / сплитовать.
- `POST|DELETE /api/treasury/adjustments[/:id]` — корректировки.

## 6. UI (макеты согласованы)

Раздел **«Касса»** (плитка только у owner, синтетическая инъекция как у finance/training):
- Остаток на счёте (якорь, редактируемый).
- **Конверты бюджета** — «можно потратить на закуп N ₽ · на хоз M ₽» (накопление с переносом), кнопка «внести факт закупки».
- Полоса корзин (ФОТ / закуп / хоз / резерв / свободно).
- **Копилки под платежи** — главный вид: прогресс «накоплено / нужно», % и статус (зелёный/жёлтый/красный).
- **Календарь месяца** — вторичный вид: метки платежей + подсветка напряжённых дней.
- Подсказка «сегодня отложи N · свободно M».
- У красного платежа — кнопки: перенести / сплит / урезать.
- Быстрое добавление: платёж и факт трат (4 категории: продукты / хозка / личные / прочее).
- Настройки: ставки `purchase_pct`/`household_pct`, подушка.
- Линия-прогноз из первого макета **отклонена** (менее информативна).

## 7. Этапы

- **Фаза 1 (MVP):** таблицы `treasury_balance` + `treasury_payments` + `treasury_settings` + `treasury_spend`;
  раздел owner-only; ввод остатка и ставок; **конверты закуп/хоз с накоплением и переносом** + факт-ввод трат;
  список платежей + копилки + флаг «обеспечен/впритык/риск» по прогнозу `FCF`. Без авто-подсказок.
  Параллельно — **консолидация Финансов** (см. §9). Цель — **месяц сверять прогноз с реальностью**.
- **Фаза 2 (на проде):** «сколько откладывать в день» (perDay) + «свободно сегодня» (freeToday/todayEarmark) +
  полоса корзин (allocation: ФОТ/закуп/хоз/резерв/свободно). Подушка уже учтена в reservableNow.
  ОСТАЛОСЬ из Фазы 2: история ставок по периодам (пока конверты копят от accrual_start по текущей ставке).
- **Фаза 3 (на проде):** авто-подсказки у платежей — перенести (ближайшая обеспеченная дата по FCF),
  сплит (эндпоинт `/payments/:id/split`: «к сроку» + «остаток позже»), «урезать» (информац. список).
  **Повторяющиеся платежи** — отметка «ежемесячно», авто-создание след. месяца при оплате (`recurring`).
  **Календарь месяца** — вторичный вид с метками платежей по статусу (фронт из payments).
  ОСТАЛОСЬ: история ставок по периодам; работа с долгами/профицитом (приоритеты).

## 8. Открытые вопросы / риски

- **Garbage in, garbage out:** точность зависит от честного ввода остатка и `operating_pct`.
  Поэтому Фаза 1 — сначала проверка прогноза на факте.
- Скоуп-крип в сторону казначейства/ERP — держать раздел простым и owner-only.

### Решено (2026-06-12)
- Операционка — **приоритетный водопад**: ФОТ (из графика) → закуп `purchase_pct` ≤30% → хоз `household_pct` ≤5%
  → резерв → свободно. НЕ один плоский %.
- **Конверты** закуп/хоз: накопление от факта, перенос (не обнуляются), списание ручным фактом трат.
- Факт трат — 4 категории: продукты(кухня+бар) / хозка / личные / прочее(форс-мажор).
- Категории платежей — фикс список + свободное имя. Подушка — фикс сумма. Прогноз выручки — `predictRevenue`.
- Долг — обычная точка платежа с низким приоритетом «по возможности» (отдельной логики в Фазе 1 нет).
- Раздел — **строго owner** (не manager).
- Ставки `purchase_pct`/`household_pct` владелец меняет периодически.

## 9. Консолидация раздела «Финансы» (руководитель)

Новая «Касса» поглощает часть старого раздела. Решение:
- **P&L · прогноз от выручки — спрятать** (не используется, данных не хватает).
- **Постоянные платежи — спрятать** (дублируется точками платежей Кассы).
- **Закрытия смен (детально) — оставить.**
- **Выручка прогноз — оставить** (Касса переиспользует `predictRevenue`).
- **«Внести расход» — заменить** на факт-ввод Кассы по 4 категориям (завязан на конверты).
- Код прятать через `getServices`/меню, не удалять (вернуть легко). `finance_fixed`/`finance_expenses` пока
  не трогаем в БД — только убираем из UI; миграции на снос — позже, если приживётся.
