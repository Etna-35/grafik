# Личный кабинет Etna / no-money-no-honey

## Цель

Построить полноценный корпоративный личный кабинет сотрудника, где вход выполняется по PIN-коду, а доступные разделы зависят от конкретного сотрудника и его роли.

Текущий календарь `index.html` остается рабочей опубликованной версией графика. Новый сервис не должен ломать эту версию: перенос календаря в backend выполняется отдельным этапом.

## Выбранная связка

Основная архитектура:

- `Postgres` - единая база данных.
- Небольшой backend - авторизация, права доступа, бизнес-логика, API.
- `NocoDB` - внутренняя админка для руководителя.
- Frontend ЛК - интерфейс сотрудника и руководителя.

Важно: `NocoDB` не является backend для сотрудников. Это админ-панель поверх базы. Все правила доступа сотрудников должны проходить через backend.

## Роли компонентов

### Postgres

Хранит:

- сотрудников;
- PIN-коды в виде хешей, без открытого хранения;
- роли и доступные сервисы;
- график смен;
- выплаты, плановые дни зарплаты, оценки сотрудников;
- задания и дедлайны;
- закрытия смен;
- аудит действий.

### Backend

Отвечает за:

- вход по PIN-коду;
- выдачу сессии;
- проверку прав;
- API для календаря, закрытия смены и будущих сервисов;
- правила видимости данных;
- ограничения, например сотрудник может уменьшить время окончания своей смены, но не увеличить;
- запись аудита;
- интеграции с Telegram, Google Sheets или другими внешними сервисами, если они останутся нужны.

Рекомендуемый технический стек backend:

- `Node.js + TypeScript`;
- `Fastify` как легкий HTTP/API сервер;
- `Drizzle ORM` или SQL migrations для Postgres;
- `Zod` для проверки входящих данных;
- `argon2id` для хеширования PIN;
- httpOnly secure cookie для сессии.

### NocoDB

Нужно использовать для управленческой работы:

- сотрудники;
- роли;
- доступные сервисы сотрудника;
- справочники;
- ручные корректировки;
- просмотр таблиц графика, выплат, оценок, заданий и закрытий смен.

Ограничение: структуру таблиц меняем миграциями backend, а не вручную через NocoDB. Через NocoDB редактируем данные.

### Frontend

Единая логика входа:

1. Пользователь открывает сайт.
2. Сразу видит ввод PIN-кода.
3. Backend определяет сотрудника.
4. Пользователь видит только доступные ему сервисы.
5. Каждый сервис открывается отдельным URL.

## Домены и адреса

Целевая схема:

- `https://lk.no-money-no-honey.ru` - вход и центральный личный кабинет.
- `https://grafik.no-money-no-honey.ru` - график смен.
- `https://smena.no-money-no-honey.ru` - закрытие смены.
- `https://api.no-money-no-honey.ru` - backend API.
- `https://admin.no-money-no-honey.ru` - NocoDB, только для руководителя.

Для MVP можно временно сделать все разделы путями одного домена:

- `/`
- `/grafik`
- `/smena`
- `/admin`

Но архитектуру сразу проектируем так, чтобы потом без переписывания развести сервисы по поддоменам.

## Почему уже не GitHub Pages

GitHub Pages подходит для текущего статического календаря, но не подходит для полноценного личного кабинета, потому что нужны:

- серверная авторизация;
- общая база данных;
- скрытие чужих данных;
- роли и доступы;
- backend-правила;
- серверные интеграции;
- безопасное хранение секретов.

Поэтому новый ЛК должен жить на сервере или PaaS, где можно запустить backend, Postgres и NocoDB.

## Минимальная серверная схема

Рекомендуемый MVP-деплой:

- один VPS;
- Docker Compose;
- контейнер `postgres`;
- контейнер `api`;
- контейнер `nocodb`;
- `Caddy` или `Nginx` как reverse proxy с HTTPS;
- ежедневные backup базы.

## Базовые правила безопасности

- PIN-коды не хранить открытым текстом.
- PIN проверяет только backend.
- Сессия хранится в secure httpOnly cookie.
- Сотрудник не получает из API чужие выплаты, оценки и задания.
- Руководитель видит все данные.
- NocoDB закрыт отдельным доступом и не используется сотрудниками.
- Все действия, влияющие на деньги, график и оценки, пишутся в audit log.

## Статус текущего календаря

Текущий опубликованный календарь:

- файл: `index.html`;
- данные: `localStorage`;
- backup: JSON из интерфейса;
- деплой: GitHub Pages;
- публичная ссылка: `https://etna-35.github.io/grafik/`.

На этапе переноса в новый сервис нужно сделать импорт существующего JSON-бэкапа в Postgres.
