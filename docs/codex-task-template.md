# Шаблон постановки задачи для Codex

Зачем: чтобы даже более дешёвая/ранняя модель Codex понимала задачу и проект без домысливания. Принцип: точная формулировка + ссылки на доки = меньше нагрузки на «соображалку» модели. Claude помогает превратить идею в такую постановку (профессиональные формулировки, нюансы), но не диктует реализацию — код пишет Codex.

## Шаблон

```
ЗАДАЧА: <короткое название>

КОНТЕКСТ (прочитай перед работой):
- AGENTS.md, DECISIONS.md, STATE.md
- <конкретный бриф, напр. docs/codex-brief-shift-close.md>

ЦЕЛЬ: <что и зачем, 1–2 предложения>

РАМКИ / ОГРАНИЧЕНИЯ:
- Не трогать: <что не менять — напр. сгенерированный код смежных модулей, data-model без необходимости>
- Роль-скоуп — в backend; стиль — как в проекте; не усложнять (2–5 одновременных пользователей).
- <доменные нюансы конкретной задачи>

КРИТЕРИИ ГОТОВО:
- <как понять, что сделано: поведение, что появилось, что считается>

ПОРЯДОК:
1. Сначала покажи план и (если меняется модель) изменения схемы — дождись ок.
2. Потом код, мелкими шагами.
3. Если поменялись решения/состояние — допиши в DECISIONS.md / STATE.md.
```

## Правила выбора модели (экономия токенов)
- Дешёвая/ранняя модель — для мелких, чётко очерченных задач: добавить поле/экран/запрос, поправка по готовой спеке, локальная правка.
- Думающая модель — для неоднозначного, архитектурного, кросс-модульного и отладки.
- Всегда: держать задачу мелкой и хорошо описанной — это сильнее всего влияет на качество дешёвой модели.

## Пример (хорошая постановка)

```
ЗАДАЧА: Тест и доводка раздела «Задачи»

КОНТЕКСТ: AGENTS.md, DECISIONS.md, STATE.md, docs/codex-brief-tasks.md

ЦЕЛЬ: Довести «Задачи» до рабочего состояния (раздел доделан, но не протестирован).

РАМКИ:
- Задачи ≠ оценки (цветные точки) — не смешивать.
- Роль-скоуп: сотрудник видит свои/общие, руководитель — все.
- Маркер задачи на календаре не должен конфликтовать с точками оценок.

КРИТЕРИИ ГОТОВО:
- Создание/назначение/смена статуса работают; фильтры Все/Мои/Сегодня/Готово; в ЛК сотрудника видны его задачи; мягкое удаление.

ПОРЯДОК: план → код мелкими шагами → при изменениях обновить STATE.md.
```
