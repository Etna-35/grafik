# Бриф Codex · Обучение / База знаний

## Задача

Сделать в корпоративном кабинете Etna раздел `Обучение` как внутреннюю базу знаний для сотрудников.

На первом этапе основной сценарий — официанты: новые сотрудники проходят `Маршрут стажера`, действующие сотрудники используют раздел как справочник по стандартам, меню, сервису и внутренним правилам.

## Граница раздела

`Обучение / База знаний` — это справочник и маршрут обучения внутри ЛК.

`Квиз / Аттестация` — отдельный сервис на `https://quiz.no-money-no-honey.ru/`, но на том же backend, той же БД и тех же PIN-аккаунтах. Подробности квиза описаны отдельно: `docs/codex-brief-quiz.md`.

## Продуктовая логика

- Раздел находится внутри ЛК и открывается через плитку `Обучение`.
- Доступ выдается через существующие индивидуальные права сотрудника.
- По умолчанию доступ нужен официантам, но нельзя жестко привязывать раздел только к роли: руководитель должен иметь возможность выдать доступ вручную.
- Контент хранится в Postgres и редактируется без правки кода: через NocoDB и/или внутреннюю админку.
- Базовая структура: 7-10 глав обучения плюс `Маршрут стажера` по дням.
- Стартовый источник контента: Google Doc `Библия` от пользователя. Документ разобран как 1 модуль `База знаний официанта`, 7 глав и 5 дней маршрута стажера.
- Текст методички не хранить в публичном git. Импорт делать напрямую из Google Doc в Postgres через импортёр, после чего источником правды становится БД.
- Каждая глава может иметь вложения: методички, артефакты, ТТК, регламенты, изображения, PDF или ссылки.
- Сотрудник может отметить главу как `прочитано`.
- Руководитель видит прогресс чтения и активность сотрудников.

## Данные

Предлагаемые сущности:

- `training_modules` — крупный раздел базы знаний.
- `training_chapters` — глава внутри раздела.
- `training_attachments` — вложения к главам: методички, ТТК, файлы, ссылки.
- `training_routes` — маршрут стажера.
- `training_route_days` — день маршрута стажера.
- `training_route_items` — конкретные главы/действия на день маршрута.
- `training_assignments` — кому назначен модуль или маршрут.
- `training_read_marks` — отметка сотрудника, что глава прочитана.

Хранить структурированно, чтобы потом строить дашборд без разбора текста.

## Права

- Рядовой сотрудник видит только назначенные ему материалы и свои отметки.
- Руководитель/менеджер видит все материалы и прогресс сотрудников.
- Редактирование контента в MVP допустимо через NocoDB, но схема БД создается только backend-миграциями.
- Нельзя показывать сотруднику чужой прогресс.

## Интерфейс сотрудника

- Плитка `Обучение` в ЛК.
- Список назначенных разделов и маршрутов.
- Прогресс по маршруту стажера.
- Главы с коротким описанием.
- Вложения к главе.
- Кнопка `Прочитано`.
- Ссылка на аттестацию по теме, если к главе привязан квиз.

## Интерфейс руководителя

- Список сотрудников и прогресс обучения.
- Сводка: сколько глав, сколько сотрудников с доступом, сколько завершили, средний прогресс.
- Видно по каждому сотруднику: прочитано / всего, процент, последняя активность.
- Редактирование контента через NocoDB/админку без изменений кода.

## API

Минимальный набор:

- `GET /api/training/init`
- `POST /api/training/chapters/{id}/read`

Админские CRUD-операции для контента можно сначала выполнять через NocoDB, если схема БД создана backend-миграциями.

## Не делать в MVP

- Отдельную авторизацию для обучения.
- Отдельную БД.
- Сложную LMS с сертификатами, SCORM, видеоаналитикой и тяжелыми редакторами.
- Внешние сервисы, которые могут не работать из РФ.

## Критерии готовности MVP

- В ЛК появляется раздел `Обучение` только у сотрудников с правом доступа.
- Контент хранится в Postgres и редактируется без кода.
- Сотрудник может открыть главу, увидеть вложения и отметить `прочитано`.
- Руководитель видит прогресс чтения и активность.
- Рядовой сотрудник не видит чужой прогресс.
