# График смен · Etna — проектный документ

> Контекст для продолжения работы в Codex / Claude Code.
> Версия документа основана на фактическом коде `index.html` (≈1188 строк), а не только на словесном описании.

---

## 1. Описание проекта

**Что это.** «График смен · Etna» — одностраничное веб-приложение для составления помесячного графика смен сотрудников ресторана-бара Etna (Вологда) и расчёта ФОТ.

**Где живёт.**
- Публично: `https://etna-35.github.io/grafik/`
- Локально: `index.html` (один файл).

**Стек и характер.**
- Чистый HTML + CSS + ванильный JS в одном файле, без сборки, без фреймворков, без зависимостей рантайма. Код обёрнут в IIFE со `"use strict"`.
- Шрифты Manrope + Playfair Display (Google Fonts). Фирменный бордовый `#8f2433`.
- Оформлено как PWA-lite: inline-манифест, `apple-touch-icon`, `theme-color`, `display:standalone` — можно «добавить на экран».
- **Бэкенда нет.** Всё состояние хранится в `localStorage` конкретного браузера. Синхронизации между устройствами нет. Есть ручной JSON-бэкап/восстановление.

**Назначение в продукте.** Сейчас это автономный инструмент владельца. В дорожной карте он должен стать разделом `grafik` внутри корпоративного личного кабинета `no-money-no-honey` (PIN-вход, роли доступа, общая база, персональный вид для каждого сотрудника).

## 2. Техническая справка (для агента)

### 2.1. Модель данных (`state`)

Хранится целиком одним JSON в `localStorage` под ключом **`etna_schedule_v1`**. Сохранение дебаунсится на 350 мс.

```js
state = {
  employees: [
    // почасовые роли (cook / bar / waiter):
    { id, name, role, hours, rate },          // hours — часы по умолчанию, rate — ₽/час
    // мойщицы (dish):
    { id, name, role, fixed, fixedWe }        // см. примечание ниже — поля по факту не используются
  ],
  shifts: {
    "YYYY-MM-DD": { [empId]: value }          // value перегружен, см. ниже
  },
  marks: {
    "YYYY-MM-DD": { task: Boolean, pay: [empId, ...] }
  },
  year, month,                                // текущий просматриваемый месяц (month 0-индексный)
  showMoney: Boolean                          // глобальный тумблер показа денег
}
```

### 2.2. Ключевые семантические нюансы (важно при миграции)

- **`shifts[dk][eid]` перегружен по смыслу:**
  - почасовая роль → число **часов** смены;
  - мойщица (`dish`) → **сумма в рублях** за эту дату.
  Оплата считается в `payOf(emp, value)`: для почасовых `rate * value`, для мойщиц — `value` напрямую.
- **`fixed` / `fixedWe` — мёртвые поля.** Создаются в `defaultState()` и нормализуются в `migrate()`, но в расчётах не участвуют (оплата мойщицы берётся из значения смены). При рефакторинге их стоит либо задействовать осмысленно, либо удалить.
- **Метки даты (`marks`):**
  - `pay` — массив id сотрудников, у кого в этот день зарплата (бейдж «₽», при >1 показывает счётчик). Это **визуальная отметка**, статуса «выплачено/не выплачено» нет.
  - `task` — булево: на дату стоит дедлайн (бейдж «☆»). Полноценной сущности «задача» нет, только флаг на дате.
  - Метки монохромные, не emoji.

### 2.3. Роли

```js
ROLES = {
  cook:   { label:"Повар",    model:"hourly" },
  bar:    { label:"Бармен",   model:"hourly" },
  waiter: { label:"Официант", model:"hourly" },
  dish:   { label:"Мойщица",  model:"fixed"  }
}
ROLE_ORDER = ["cook","bar","waiter","dish"]   // порядок сортировки колонок
```

### 2.4. Деньги и план выручки

- Константы нормы ФОТ: `FOT_MIN_RATE = 0.23`, `FOT_MAX_RATE = 0.28`.
- `planRange(fot)` = диапазон выручки, при котором ФОТ попадёт в 23–28%: от `fot/0.28` до `fot/0.23`, в формате «к» (например `35к-40к`).
- Пресеты оплаты мойщицы: `[3000, 4000, 6000, 8000]` + «своя».
- Сводка считает: ФОТ по дню, ФОТ за месяц, зарплату по сотруднику, итоги по ролям, дней со сменами, средние за рабочий день и за календарный день.

### 2.5. Взаимодействия (UI)

- **Тап по клетке** — поставить/снять смену. Для мойщицы тап открывает шит выбора оплаты.
- **Долгое нажатие (460 мс, touch и mouse)** — открыть шит с индивидуальными часами/оплатой смены.
- **Тап по дате (левая колонка)** — открыть шит меток (зарплата / дедлайн).
- Тумблер «₽ Деньги: вкл/выкл», переключение месяцев, очистка месяца (с confirm).

### 2.6. Экспорт и бэкап

- **PNG:** `buildImage()` рисует график на `<canvas>` (своя палитра `HEX`, скруглённые прямоугольники, обрезка текста). На устройствах с Web Share API — шеринг картинки; на десктопе — скачивание файлом `grafik-etna-YYYY-MM.png`.
- **JSON-бэкап:** `exportBackup()` → файл `etna-grafik-YYYYMMDD.json`. `importBackup()` валидирует (`Array.isArray(employees)`), спрашивает подтверждение, прогоняет `migrate()`.

## 3. Текущий функционал (что уже работает)

- Помесячный график: сотрудники — колонки, даты — строки, переключение месяцев.
- Сортировка сотрудников по ролям, подсветка выходных и текущего дня.
- Колонка `Σ` — число сотрудников в смене по каждому дню (coverage).
- Смены: тап (вкл/выкл), долгое нажатие (свои часы), для мойщиц — выбор суммы за смену по дате.
- CRUD сотрудников: добавить, удалить (со сменами), переименовать, сменить роль; для почасовых — часы по умолчанию и ставка ₽/час.
- Очистка месяца.
- Деньги: тумблер вкл/выкл, ФОТ по дню/месяцу, зарплата по сотруднику, итоги по ролям, план выручки.
- Метки на даты: ₽ (день зарплаты, с счётчиком) и ☆ (дедлайн), монохромные.
- Экспорт картинки графика (с метками, ФОТ, планом), скачивание PNG на десктопе.
- JSON-бэкап и восстановление с миграцией схемы.

## 4. Ограничения текущей версии

- Нет PIN-входа, личных кабинетов сотрудников, ролей доступа.
- Нет общей БД и синхронизации между устройствами — данные привязаны к браузеру.
- Нельзя безопасно показать сотруднику только его данные.
- Нет сущности «задача» — только флаг дедлайна на дате.
- Нет статусов выплат — метка ₽ чисто визуальная.
- Нет фактического времени начала/окончания смены — только длительность в часах.
- Сотрудник не может сам редактировать окончание смены.
- Перегруженное значение смены (часы vs рубли) и мёртвые поля `fixed`/`fixedWe` мешают чистому переносу в БД.

## 5. Задачи (бэклог)

Сгруппировано по уровню риска и зависимостям. Метки: **[S]** мелкая, **[M]** средняя, **[L]** крупная.

### A. Чистка и подготовка к переносу (без смены архитектуры)
- **[M]** Развести смысл смены: вместо перегруженного числа хранить типизированный объект, напр. `{ kind:"hours", hours:12 }` / `{ kind:"pay", pay:6000 }`. Обновить `getShift/setShift/payOf` и `migrate` (со старой схемы).
- **[S]** Убрать или осмысленно задействовать `fixed`/`fixedWe`.
- **[M]** Вынести доступ к данным в отдельный модуль-«хранилище» с интерфейсом `getShift/setShift/getMark/...`, чтобы потом подменить `localStorage` на API без переписывания рендера.
- **[S]** Зафиксировать версию схемы в самом `state` (`schemaVersion`) — упростит миграции.

### B. Быстрые продуктовые улучшения (в рамках localStorage)
- **[S]** Статусы выплат: метка ₽ → состояние «запланировано / выплачено».
- **[M]** Фактическое время смены (начало/конец), а не только длительность; длительность считать автоматически.
- **[M]** Сущность «задача»: заголовок, исполнитель, дедлайн, статус — вместо голого флага ☆.
- **[S]** Экспорт месяца в CSV (для сверки с другими таблицами Etna).

### C. Бэкенд и общая база (ядро миграции)
- **[L]** Выбрать стек хранения (см. раздел 7 — открытое решение).
- **[L]** Спроектировать схему БД (черновик в разделе 6).
- **[L]** API/слой синхронизации: чтение/запись графика, сотрудников, меток, выплат.
- **[M]** Перенос текущих данных из JSON-бэкапа в БД (импортер).

### D. Личный кабинет `no-money-no-honey` и роли
- **[L]** PIN-вход и главная навигация (`no-money-no-honey/`).
- **[L]** Роли доступа: владелец/админ видит всё; сотрудник видит общий график, но свою зарплату/выплаты/задачи/правки — только в рамках прав.
- **[M]** Раздел `grafik` — текущий календарь поверх общей базы.
- **[L]** Раздел `shift-close` — закрытие смены.

### E. Интеграции (опираясь на существующую экосистему Etna)
- **[M]** Связка с Telegram-ботом и Google Sheets (уже есть в других проектах Etna).
- **[S]** Сопоставление ФОТ из графика с фактической выручкой/закупом.

## 6. Черновик схемы БД (на этап C)

Ориентир, не финал. Развести перегруженные сущности заранее.

- **users** — `id, name, role(owner|admin|staff), pin_hash, employee_id?`
- **employees** — `id, name, role(cook|bar|waiter|dish), pay_model(hourly|fixed), default_hours, hourly_rate, active`
- **shifts** — `id, employee_id, date, hours?, pay_amount?, start_time?, end_time?` (часы и сумма — отдельные поля, не одно перегруженное)
- **tasks** — `id, title, assignee_employee_id, due_date, status`
- **payments** — `id, employee_id, date, amount, status(planned|paid)`
- **settings** — нормы ФОТ, пресеты, и т.п.

Контракт данных при экспорте/синхронизации стоит зафиксировать как версионированный JSON, совместимый с текущим бэкапом (для бесшовного импорта).

## 7. Зафиксированное решение по стеку

Для полноценного личного кабинета выбран стек:

- **Postgres** — основная база данных.
- **Небольшой backend** — PIN-вход, права доступа, бизнес-логика, API графика, выплат, задач и закрытия смены.
- **NocoDB** — внутренняя админка руководителя поверх базы, не публичный интерфейс для сотрудников.
- **Docker Compose на VPS в РФ** — MVP разворачивается на VPS Beget, чтобы сервис работал из РФ без VPN.

Backend владеет схемой БД и миграциями. Сотрудники не ходят напрямую ни в NocoDB, ни в Postgres.

## 8. План проекта (этапы)

1. **Фаза 0 — Чистка (A).** Типизировать смену, убрать мёртвые поля, вынести слой данных, добавить `schemaVersion`. Цель: текущее приложение работает так же, но готово к подмене хранилища.
2. **Фаза 1 — Продуктовые улучшения (B).** Статусы выплат, время смены, сущность «задача», CSV-экспорт. Всё ещё на localStorage — быстро и без бэкенда.
3. **Фаза 2 — Бэкенд (C).** Docker Compose → Postgres → API → миграции → импортер из JSON. На выходе: график живёт в общей базе, синхронизация между устройствами.
4. **Фаза 3 — ЛК и роли (D).** PIN-вход `no-money-no-honey`, роли, персональный вид сотрудника, раздел `grafik` поверх базы.
5. **Фаза 4 — Закрытие смены и интеграции (D/E).** `shift-close`, связка с Telegram/Sheets, сверка ФОТ с выручкой.

Каждую фазу можно отдавать агенту автономно: фаза 0 разблокирует все остальные, поэтому делать её первой.
