# План правок после тестов — 2026-06-11

10 пунктов от владельца после тестирования. Ниже — план в **логическом порядке программирования**:
сначала анализ (без кода), потом изолированные мелкие правки, затем правила бэкенда, фича
«команда в прогрессе», фундамент единиц/форматирования и в конце — крупная телеграм-фича
(она зависит от всего форматирования). Каждая задача: файлы, подход, зависимости, риск, можно ли
отдать дешёвой модели.

Принцип порядка: **данные → бэкенд → фронт**; общий код раньше зависимых фич; рискованное и
кросс-модульное — в конце; мелкое независимое — вперёд (быстрые победы, не ломают друг друга).

Сквозные правила (CLAUDE.md §3–§8): цвета только через CSS-переменные; текст на золоте тёмный;
без emoji в UI (в Telegram emoji допустимы — это не UI); имена не уникальны; статика запекается в
образ (после `apps/web` — scp + `docker compose build api`); импортёры вручную; household не трогать
в importProductCatalog; миграции только вперёд, идемпотентно. Деплой строго по §4.

## Фаза 0 — Аналитика (без кода)

### П3. Прогноз скорости прокачки уровней (3 сценария)
Деливерабл — расчёт, не код. Константы (`progress.ts`): уровень = **100 очков**; стаж **+30/мес**;
задание рук-ля +30, ролевое +10, похвала коллеги +10 / рук-ля +20, «сказал спасибо» +5 (лимит 5/день),
план +50, заявка +5, передача +5, смена +10, глава +5, тест +15, аттестация +40, челлендж +20 (лимит 2/сут),
цель продаж +20.

**Расчёт (готов, можно показывать владельцу):**
- **Рядовой (пассивный):** по сути только стаж +30/мес + редкие похвалы → ~30–50 очк/мес →
  **1 уровень за 2–3 месяца** (чистый стаж: 100/30 ≈ 3,3 мес/уровень).
- **Активный:** стаж 30 + закрытия смен (~10×10=100) + ежедневная передача (~150/мес) + похвалы/задания +
  разовый «контент» обучения (главы ~80, тесты ~240, аттестации 120 = **~440 очков единоразово за 1–2 мес**).
  → первые 1–2 месяца очень быстро (**+3–6 уровней**), дальше выходит на **~1–2 уровня/мес**.
- **Фармит по максимуму:** дневные потолки складываются — челлендж +40/сут (2×20), «спасибо» +25/сут
  (5×5), заявка (после П2) +20/сут, передача +5, смена +10 ≈ **+100 очк/сут = ~1 уровень/день** на потоке,
  плюс разовый бурст контента. → реально **уровень 30+ за месяц**.

**Вывод для П2 и баланса:** главные «фермы» — `challenge_test` (+40/сут) и `praise_giver` (+25/сут);
`requisition_sent` сейчас **без лимита** (его и чинит П2). Рекомендация: после П2 проверить, не нужен ли
аналогичный дневной потолок на передачи (`handover_sent`) — тот же вектор дробления.

## Фаза 1 — Изолированные фронт-правки (низкий риск, без зависимостей, дешёвая модель)

### П1. Разрешить ноль в форме закрытия смены (но требовать ручной ввод)
**Корень бага найден:** `moneyField()` (app.js ~3485) рендерит `value="${value ? Number(value) : ""}"` —
при значении `0` оно falsy → поле очищается. На `change` форма пересобирается и ре-рендерится → введённый
0 пропадает, а `firstMissingShiftField()` (читает DOM, ~3779) снова видит пусто → «заполните поле». Замкнутый
круг: ноль ввести невозможно.
**Подход (различать «не трогали» = null и «ввёл 0»):**
1. `moneyField`: `value="${value == null ? "" : Number(value)}"` (ноль показывается как «0»).
2. `bindShiftClosingForm` input/change (~3549, 3552): хранить `integerOrNull(input.value)` **без `|| 0`**
   (null при пустом, 0 при «0»). `integerOrNull` уже корректен (пусто→null, «0»→0).
3. `shiftClosingFormFrom` (~3661): денежные поля инициализировать **null** (не 0/openingCashExpected),
   чтобы форма стартовала пустой и требовала ручного ввода каждого поля.
4. `collectShiftClosingForm` (~3690): `next[field] = integerOrNull(...)` (оставить null). Payload/preview уже
   используют `Number(form.x || 0)` — null→0 безопасно.
**Готово:** новое поле стартует пустым; ввод «0» сохраняется и проходит валидацию; payload содержит 0.
**Файлы:** `apps/web/app.js`. **Риск:** низкий. **Проверка:** превью формы (нужен бэкенд) / на проде владельцем.

## Фаза 2 — Правило бэкенда (без схемы, дешёвая модель)

### П2. Лимит начисления % за заявку: не более 2 раз за 12 часов
**Сейчас:** `requisitions.ts` POST (~211) всегда `awardPoints(user, "requisition_sent", ... reqId)`. Дедуп
только по ref (одна заявка = одно начисление), но дробление даёт сколько угодно начислений.
**Подход:** перед `awardPoints` посчитать события `requisition_sent` этого сотрудника за последние 12 ч;
если уже ≥2 — заявку создаём, но очки **не** начисляем.
```sql
SELECT COUNT(*) FROM progress_events
WHERE employee_id=$1 AND kind='requisition_sent' AND created_at > now() - interval '12 hours'
```
Реализовать хелпером в `progress.ts` (напр. `countRecentAwards(employeeId, kind, hours)`) и вызвать в
requisitions перед начислением. **Связь с П10:** после П10 заявка станет «правкой общего сообщения», но
очковый потолок всё равно нужен (вектор дробления). **Готово:** 3-я заявка за 12 ч не даёт очков, но создаётся.
**Файлы:** `apps/api/src/progress.ts`, `requisitions.ts`. **Риск:** низкий.

## Фаза 5 — Крупная телеграм-фича (схема + бэк + фронт; в конце; включает П6)

### П10 (+П6). Редактирование заявок сотрудниками + единое общее сообщение в Telegram
**Суть (от владельца):** первый создавший заявку «на завтра» формирует **стартовое сообщение** в общем чате;
последующие сотрудники добавляют свои позиции из ЛК — но в Telegram **одно общее сообщение**, которое **бот
редактирует** (а не шлёт новое). Новая позиция помечается **➕ и жирным**, удалённая — **➖ и зачёркнутым**.
Всё за день копится в одной заявке. П6: всё это под **сворачиваемой цитатой** (`<blockquote expandable>`),
чтобы не было полотна в рабочем чате.

**Ключевое решение — ключ группировки = календарный день МСК создания** (`created_at::date` в Europe/Moscow).
Заявки «на завтра» собираются в течение дня → один батч = заявки одного дня. Следующий день → новое сообщение.
*(Согласовать с владельцем, если хочется привязки к «дате назначения», а не дню оформления.)*

**Порядок реализации (внутри задачи):**
1. **Миграция 033** (`033_requisition_tg_messages.sql`, идемпотентно): таблица
   `requisition_tg_messages(batch_date date PRIMARY KEY, chat_id text, message_id text, snapshot jsonb,
   updated_at timestamptz default now())`. `snapshot` — последний опубликованный набор строк (для диффа
   ➕/➖). Обновить `docs/data-model.md` и CLAUDE.md §6.
2. **telegram.ts:** добавить `editMessageText(chatId, messageId, text)` (метод `editMessageText`, parse_mode
   HTML). Учесть лимит 4096 симв. на сообщение (при переполнении — сворачивать менее важное / обрезать хвост).
3. **requisitions.ts:** заменить per-заявочный `notifyRequisition` на `syncDailyRequisitionBatch(batchDate)`:
   - агрегировать ВСЕ непрорезанные (`status<>'rejected'`) строки заявок за `batch_date`;
   - построить тело: `<blockquote expandable>` → СРОЧНО сверху, затем по категориям, единицы через `unitShort`
     (П7), автор у позиции опц.;
   - сравнить с `snapshot`: новые позиции → `➕ <b>…</b>`, исчезнувшие → `➖ <s>…</s>` (показываем зачёркнутыми
     один цикл/до следующего пересчёта);
   - если запись батча есть → `editMessageText(message_id)`, иначе `sendMessage` и сохранить `message_id`;
   - записать новый `snapshot`.
   Вызывать sync при: создании заявки, правке автором, удалении строки/заявки, смене статуса менеджером.
4. **Редактирование автором (бэк):** эндпоинты для автора, пока заявка в статусе new/accepted:
   - `PUT /api/requisitions/:id/lines` — заменить набор строк (валидация как в `normalizeLinesForUser`);
   - `DELETE /api/requisitions/:id` (или строки) — мягко/жёстко, затем sync с пометкой ➖.
   Скоуп: только свой `author_id`. После любой мутации — `syncDailyRequisitionBatch`.
5. **Фронт «Мои заявки»:** на своей заявке (new/accepted) — режим правки (переиспользовать корзину/степперы):
   изменить кол-во, добавить/удалить позиции, сохранить → дёргает PUT и ре-sync. UI в стиле бренда, без emoji.
6. **П2-взаимодействие:** очки за «отправку» — по-прежнему через лимит П2 (правка не должна множить очки).

**Готово:** в общем чате один редактируемый блок-цитата за день; добавления ➕жирным, удаления ➖зачёркнуто;
сотрудник правит свою заявку из ЛК, бот обновляет то же сообщение. **Файлы:** `migrations/033_*.sql`,
`apps/api/src/telegram.ts`, `requisitions.ts`, `apps/web/app.js` (+styles), `docs/data-model.md`, `CLAUDE.md`.
**Риск:** высокий (схема + внешний API + дифф-состояние + новый UI). **Модель:** думающая. Делать
пошагово: миграция → editMessageText → sync без диффа (просто перерисовка) → дифф ➕/➖ → правка автором → UI.
