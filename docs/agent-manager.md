# Агент-руководитель — стратегия и автоматизация заданий/наград

> Архитектурная заметка (бэклог). Идея владельца от 2026-06-07: использовать Claude как помощника
> руководителя — формулировать стратегию, готовить задания (общие/индивидуальные) и предложения по наградам,
> а после утверждения — публиковать их в систему. Здесь зафиксированы подход, что уже есть, чего не хватает и план.
> Сначала прочитать `CLAUDE.md` и `docs/shift-close-formulas.md`. Статус: НЕ реализовано.

## Цель
Руководитель в диалоге с Claude выстраивает стратегию → Claude предлагает задания и награды → руководитель
утверждает → система их создаёт и выставляет (через ЛК / API / MCP). Быстрее и качественнее ручного ввода.

**Главный принцип: черновик → утверждение человеком → публикация.** LLM предлагает, человек одобряет, система исполняет.

## Что уже готово в коде
- **REST API задач:** `POST /api/tasks`, `PATCH /api/tasks/:id/status`, `DELETE /api/tasks/:id` (`apps/api/src/tasks.ts`).
  Задания индивидуальные (`employee_id`) и ролевые/общие (`audience_role`).
- **Награды и очки:** премии за задачи/цели → «Выплаты» (`payroll.ts`); очки прогресса → `progress_events` (`progress.ts`).
- **Аудит** каждого действия (`audit()` в `auth.ts`) — кто/что/когда. Критично для агентского доступа.
- **Telegram-уведомления** местами заложены (план выполнен / аттестация / цель) — агентский поток в них ляжет.

## Чего не хватает (достроить)
- **Авторизация только сессия/PIN через cookie** (`auth.ts`). Нужен **служебный API-токен для агента**
  (отдельный аккаунт-актор с ограниченными правами), чтобы агент действовал от своего имени, а не под PIN владельца.
- **Состояние «черновик»** у задач/наград (draft → published) + экран утверждения.
- (Опц.) **MCP-сервер** или **панель «Помощник» в ЛК**.

## Варианты реализации (от простого к удобному)
- **A. Пакет черновиков (минимум работы — первый шаг).** Claude отдаёт список задач структурой → руководитель
  смотрит → кнопка/скрипт публикует через API. Хорошо для «раз в неделю накидать задания смене».
- **B. MCP-инструменты.** MCP-сервер с операциями проекта (`create_task`, `list_employees`, `propose_reward`).
  В чате: Claude видит сотрудников/историю, предлагает, владелец подтверждает — исполняется. Самый гибкий путь.
- **C. Панель «Помощник» в ЛК.** Пишешь цель → Claude рисует черновики инлайн → правки галочками → «Опубликовать».
  Лучший UX, больше работы.

## Награды — две развилки
1. **Детерминированные правила** (безопасно, без ИИ в деньгах): «задание выполнено → +N», «похвала → +10».
   Часть уже работает; расширять как простой движок правил. Можно автоматизировать на 100%.
2. **Награды по суждению агента** («кто заслужил премию за неделю?»): Claude **предлагает** на данных,
   но **выдачу денег подтверждает человек**. LLM не двигает деньги автономно — доверие + безопасность.

## Безопасность (обязательно до агентского доступа)
- Сделать репозиторий **приватным** + **перевыпустить токены** (уже в бэклоге `CLAUDE.md` §9).
- Служебный токен — **скоупированный** (только нужные операции), пишется в `.env`, не в git.
- Все агентские действия — через `audit()` с атрибуцией актора-агента.

## Рекомендованный первый шаг
Вариант **A**: служебный API-токен + endpoint «создать пакет задач» в статусе «черновик» + утверждение в ЛК.
Проверить удобство на нём, прежде чем вкладываться в MCP/панель. Проект маленький (≤15 чел) — не переусложнять.
