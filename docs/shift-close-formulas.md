## 1. Назначение и поток данных

Сотрудник в конце смены заполняет форму с кассовыми показателями за день. Backend считает производные
значения, пишет **одну запись на рабочую дату** в `shift_closings` и формирует три выхода:

1. **Запись в БД** (`shift_closings` + связанные таблицы) — основа для ЛК руководителя и истории.
2. **Отчёт руководителю в Telegram** (закрытый чат) — подробный, с фотографиями. См. §8 *(разбирается отдельной задачей)*.
3. **Мини-отчёт команде в Telegram** (общий чат) — краткий. См. §9 *(разбирается отдельной задачей)*.

Рабочая дата (`work_date`): смена, закрытая **до 06:00 МСК**, относится к **предыдущему** календарному дню.
✅ `currentShiftWorkDate()` в `shiftClosing.ts:995`.

## 3. Касса на открытии: подтверждение или коррекция 🟡

В начале формы показывается **расчётная сумма в кассе** (`opening_cash_expected`), перенесённая из остатка
прошлой смены (`closing_cash_actual` последней записи с `work_date < текущей` — ✅ `getOpeningCashExpected`,
`shiftClosing.ts:770`).

Сотруднику задаётся вопрос: **совпадает ли фактический остаток с расчётным?**

- **Да** → `opening_cash_actual = opening_cash_expected`. Дальнейшие расчёты идут от этого значения.
- **Нет** → сотрудник вводит фактическое значение в `opening_cash_actual`. Дальнейшие расчёты идут
  от **фактического (ручного)** значения. Расхождение фиксируется:

```
opening_cash_diff = opening_cash_actual − opening_cash_expected
```

✅ Поля `opening_cash_expected/actual/diff` уже есть. 🟡 UX «подтвердить / скорректировать» оформить явно
(сейчас в форме просто ввод факта).


## 5. Переводы — с привязкой к сотруднику 🟡

«Переводы» = клиент перевёл деньги на **личную карту конкретного сотрудника** (в физическую кассу деньги
не попадали). При заполнении поля нужно **выбрать сотрудника, которому сделан перевод, и сумму**;
возможно несколько строк (несколько сотрудников).

- В **выручку** переводы входят (§4).
- На **физический остаток наличных в кассе НЕ влияют** (см. §7 — решение владельца от 2026-06-07).
- В отчёте показываются отдельной строкой с именем получателя (§8).

🟡 Сейчас в коде `transfer_revenue` — одно число без привязки к сотруднику. Нужна отдельная таблица-разбивка
(по аналогии с `shift_closing_extra_expenses`), например `shift_closing_transfers (shift_closing_id, employee_id, amount)`.

## 6. Расходы из кассы

```
Выпл. кальяны  hookah_payout = hookah_count × hookah_rate
```
- `hookah_rate` — ставка **на сотруднике-кальянщике** (`employees.hookah_rate`), не глобальная.
- Это одновременно расход кассы И заработок кальянщика (одни деньги в двух разрезах, не дублировать).
- ✅ `shiftClosing.ts:394`, попадает в «Выплаты» кальянщика.

Доп. расходы: список строк «сумма + комментарий», в кассе вычитается `extra_expenses_total` (сумма строк).
✅ `shift_closing_extra_expenses`.

### Несколько кальянщиков в смену ✅ РЕАЛИЗОВАНО (2026-06-07, миграция 022)
Поддержаны **несколько кальянщиков** в одной смене — дочерняя таблица-список по образцу
`shift_closing_extra_expenses`:
```
shift_closing_hookah (id, shift_closing_id, employee_id, count, rate, payout, created_at)
    payout = count × rate   (rate берётся из employees.hookah_rate выбранного сотрудника)
```
- В форме (§2) — список строк «кальянщик + кол-во» (несколько). `hookah_payout` смены = `Σ payout`.
- **Индивидуальный показ в ЛК.** Каждый кальянщик видит **свои** кальяны (кол-во и заработок) в своём
  разделе «Доходы/Выплаты» — `payroll.ts` уже фильтрует по `employee_id`, после перехода на дочернюю
  таблицу это работает само (join по `shift_closing_hookah.employee_id`).
- Миграция: создать таблицу + перенести существующие записи из колонок `shift_closings.hookah_*`.
  Старые колонки можно оставить как легаси/денормализованный итог либо удалить — решить при реализации.

> Примечание: ранее «два кальянщика в смену» были в отменённых (`CLAUDE.md` §9). Это решение их **возвращает**
> по явному запросу владельца от 2026-06-07. Обновить `CLAUDE.md`/`DECISIONS.md` при реализации.

**Фолбэк, если откажемся от полной правки:** оставить одного `hookah_employee_id` + добавить **поле-комментарий**
к кальянам (вручную вписать имена получивших). Тогда индивидуального показа в ЛК не будет.

## 7. Инкассация и остаток наличных в кассе

### Инкассация (для отображения) 🟡
```
Инкассация (отчётная) = collection_amount (физ.) + Σ переводов
```
Это **только для показа** в отчёте/ЛК («сколько денег ушло из-под ответственности кассы»). На расчёт
физического остатка переводы **не влияют**.

### Остаток наличных на конец дня (расчётный) 🟡 — целевая формула
```
closing_cash_expected =
    opening_cash_actual            (касса с прошлого дня: расчётная ИЛИ ручная коррекция, §3)
  + cash_revenue                   (наличная выручка)
  − wash_cost                      (мойка)
  − hookah_payout                  (выпл. кальяны)
  − extra_expenses_total           (доп. расходы)
  − collection_amount              (ФИЗИЧЕСКАЯ инкассация, БЕЗ переводов)
```

```
Разница  closing_cash_diff = closing_cash_actual − closing_cash_expected
```
- `closing_cash_diff < 0` → **недосдача**; `> 0` → **излишек**.
- Такси (`taxi_amount`) в формулу НЕ входит (логируется отдельно).

> ⚠️ **РАСХОЖДЕНИЕ С ТЕКУЩИМ КОДОМ.** Сейчас `shiftClosing.ts:395–402` считает остаток с **переводами**:
> `... + cash_revenue + transfer_revenue − ...`. По решению владельца (2026-06-07) переводы из формулы
> остатка **убираются** — переход на формулу выше. Это доработка backend.

### Перенос на следующую смену
`opening_cash_expected` следующей смены = `closing_cash_actual` (факт) этой смены. ✅

## 8. Лимиты и флаги
- Лимит расхождения кассы = **500 ₽** (`cash_diff_limit`). ✅ захардкожен.
- Лимит такси на смену = **2000 ₽** (`taxi_limit`). ✅ захардкожен.
- ✅ `flags.cashDiffExceeded = |closing_cash_diff| > 500`, `taxiExceeded = taxi_amount > 2000`.

**Пометка кассы (решение владельца 2026-06-07):** строка статуса кассы показывается при **любом** ненулевом
расхождении (`closing_cash_diff ≠ 0`): недосдача (`< 0`) / излишек (`> 0`) с суммой |diff| (пример: «недосдача
в 10»). Лимит 500 ₽ остаётся как порог **усиленной** пометки (`cashDiffExceeded`) — напр. дополнительный акцент.

## 9. План выручки и проценты

```
План выручки (день)  revenue_plan = round( ФОТ_дня / 0.23 )
    где ФОТ_дня = SUM(schedule_shifts.pay_amount) за work_date
% выполнения          revenue_plan_percent = revenue_total / revenue_plan × 100
```
✅ `getRevenuePlan` (`shiftClosing.ts:784`), единый источник плана — модуль «График».

**Наличный план (день)** = ФОТ дня (та же сумма `SUM(pay_amount)`, без деления на 0.23).
В отчёте: `Наличные · % = cash_revenue / наличный_план`. 🟡 Сейчас в `shiftClosing.ts` наличный план
отдельно не считается — добавить (значение уже доступно как промежуточное в `getRevenuePlan`).

## 10. ЛК руководителя — дашборд 🟡

### Блок 1 — сводка сверху
- **Общая выручка за месяц** = `Σ revenue_total` по дням месяца.
  Мелким шрифтом: план месяца (`Σ revenue_plan` по дням) и % выполнения.
- **Выручка за день** (текущий день) = `revenue_total`.
  Мелким шрифтом: план дня и %.
- **Отставание / опережение плана по итогам прошедших дней.** Базовое значение — алгебраическая сумма
  отклонений по прошедшим дням месяца:
  ```
  баланс = Σ (revenue_total_дня − revenue_plan_дня)  по ПРОШЕДШИМ дням месяца
           ( > 0 — опережаем; < 0 — отстаём )
  оставшиеся_дни = число дней от сегодня до конца месяца ВКЛЮЧИТЕЛЬНО (календарные)
  ```
  Отображение зависит от знака:

  **A) Отстаём (`баланс < 0`)** — показываем, сколько нужно ДОПОЛНИТЕЛЬНО в день (сверх обычного дневного
  плана), чтобы вытянуть отставание:
  ```
  нужно_доп_в_день = |баланс| / оставшиеся_дни
  ```
  *Пример:* отставали на 5 000 ₽ каждый из 6 дней, сегодня 7 июня, осталось 24 дня (с сегодняшним):
  `30 000 / 24 = 1 250 ₽/день дополнительно`.

  **B) Опережаем (`баланс ≥ 0`)** — показываем **общую сумму опережения, БЕЗ деления на дни**:
  ```
  опережение = баланс        (одно число, как есть)
  ```
  *Пример:* 3 дня опережали по 3 000 ₽ и 3 дня отставали по 1 000 ₽:
  `+3+3+3−1−1−1 (тыс.) = +6 000 ₽ опережения`.

  - «оставшиеся_дни» — **календарные**, включая сегодня (решение владельца 2026-06-07).
  - Считается по прошедшим дням; перевыполнение и недовыполнение суммируются алгебраически (взаимозачёт).

### Блок 2 — список дней месяца
Для каждого дня:
- Выручка (`revenue_total`) и отклонение от плана (`revenue_total − revenue_plan`, либо %).
- **Фактическое поступление на расчётный счёт** (за вычетом комиссий):
  ```
  на_расч_счёт =
      (terminal_1 + terminal_2) × (1 − 0.02)      эквайринг 2% с терминалов
    + yandex_food               × (1 − 0.35)      комиссия Яндекс.Еда 35%
    + netmonet                  × (1 − 0.005)     комиссия Нетмонет 0.5%
    − taxi_amount                                  такси оплачено по терминалу (решение 2026-06-07)
  ```
  Наличные и переводы на расчётный счёт **не попадают** (наличные — касса, переводы — личные карты сотрудников).
  Такси (`taxi_amount`) — это безналичный расход по терминалу: уменьшает **только** фактическое поступление на
  расч. счёт; «Безнал итого» и «Выручка итого» он **не** меняет (такси — расход, а не уменьшение продаж).

**Комиссии (решение владельца 2026-06-07):** ставки `2% / 35% / 0.5%` остаются **константами в коде** —
меняются редко, отдельные настройки не нужны. Опционально (low-priority): спрятать редактирование под иконку
шестерёнки для редких правок, если это просто.

### Мотивационная подача метрики (для сотрудников) 🟡
Значение «нужно доп. в день» можно показывать команде (на главной/ЛК сотрудника) в мотивирующем виде, переводя
рубли в понятный ориентир — количество апсейлов:
```
N = ceil( доп_в_день / цена_апсейла )      → "N × <название апсейла>"
```
Пример текста:
> Коллеги, мы отстаём от плана — нужно дополнительно **1 250 ₽** ежедневно. Это всего **3 пирожных**.
> Действуем и закрываем цель! Лучшие 💪

- **Апсейл задаёт руководитель вручную: название + цена** (напр. «пирожное» 450 ₽, «настойка» 250 ₽).
  Хранить настраиваемую пару `{ upsell_name, upsell_price }` (одно значение, редактируется руководителем).
  Округление количества — вверх (`ceil`).
- Если идём по плану/опережаем — позитивный вариант сообщения вместо «отстаём».

- Такси — безналичный расход по терминалу: вычитается **только** из поступления на расч. счёт; выручку/безнал не меняет.
- Обязательные поля отчёта — все денежные **кроме** кальянов и доп. расходов (они опциональны).
- Переводы **не влияют** на остаток наличных (агрегат «инкассация + переводы» — только для отображения).
- Несколько кальянщиков в смену — **делаем** (дочерняя таблица + индивидуальный показ в ЛК).
- «Оставшиеся дни» — **календарные**, включая сегодня; добавка = накопленный дефицит / оставшиеся дни.
- Пометка кассы — при **любом** расхождении `diff ≠ 0`; лимит 500 — усиленная пометка.
- Комиссии расч. счёта (2% / 35% / 0.5%) — **константы в коде** (опц. шестерёнка для редких правок).
- Мотивационный апсейл — руководитель задаёт **название + цену** вручную; кол-во округляется вверх.
