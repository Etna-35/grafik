# План реализации ЛК Etna

## Принцип

Делаем сервис поэтапно. Сначала технический фундамент, потом переносим существующий календарь, затем подключаем закрытие смены и новые сервисы.

Текущий график на GitHub Pages остается рабочим до тех пор, пока новый ЛК не сможет заменить его без потери данных.

## Этап 0. Подготовка

Результат:

- зафиксирована архитектура;
- описана модель данных;
- выбрана связка `Postgres + backend + NocoDB`;
- определены домены;
- определены границы MVP.

Статус: выполнено.

## Этап 1. Инфраструктура

Что сделать:

- создать новый backend-проект;
- добавить Docker Compose;
- поднять `Postgres`;
- поднять `NocoDB`;
- поднять backend API;
- настроить `.env.example`;
- добавить healthcheck API;
- добавить миграции базы;
- подготовить reverse proxy для доменов.

Минимальный состав контейнеров:

- `postgres`;
- `api`;
- `nocodb`;
- `caddy` или `nginx`.

Результат этапа:

- локально запускается база, API и NocoDB;
- можно открыть админку;
- можно выполнить первую миграцию;
- API отвечает `200` на `/health`.

Статус: выполнено на Beget VPS. API отвечает `200` на `/api/health`, NocoDB открыт через `admin.no-money-no-honey.ru`, база и миграции работают в Postgres.

## Этап 2. Авторизация и центральный ЛК

Что сделать:

- таблицы сотрудников;
- таблицы PIN-хешей;
- вход по PIN;
- защита от перебора PIN;
- сессии;
- список доступных сервисов;
- главный экран ЛК после входа.

MVP-экран:

- 4 точки PIN;
- после входа - кнопки сервисов;
- сотрудник видит только разрешенные сервисы.

Результат этапа:

- руководитель может завести сотрудника;
- сотрудник входит по PIN;
- backend возвращает список его сервисов.

Статус: MVP выполнен. Создан первый owner `Руководитель`, PIN хранится как argon2id-хеш, вход выдаёт httpOnly cookie, backend возвращает доступные сервисы.

## Этап 3. Перенос календаря

Что сделать:

- перенести модель сотрудников;
- перенести смены;
- перенести выплаты;
- перенести плановые дни зарплаты;
- перенести оценки;
- перенести дедлайны;
- сделать импорт JSON-бэкапа из текущего календаря;
- заменить `localStorage` на API.

Правила:

- руководитель видит полный график;
- сотрудник видит общий график, но персональные деньги только свои;
- сотрудник может уменьшить окончание своей смены;
- выплаты не меняют расчетный дневной ФОТ и план выручки.

Результат этапа:

- календарь работает внутри ЛК;
- данные хранятся в Postgres;
- старый JSON можно импортировать.

Статус: основной перенос начат и уже редактируемый. Реализованы backend API `/api/schedule`, импорт JSON-бэкапа через `/api/schedule/import`, миграция полей сотрудников для графика, экран просмотра графика внутри ЛК и manager-only редактирование ячеек календаря через API.

Уже можно через ЛК:

- поставить и снять смену сотруднику;
- изменить часы или фиксированную сумму смены;
- отметить дедлайн дня;
- назначить день зарплаты конкретному сотруднику;
- внести и удалить фактическую выплату;
- поставить и снять зеленую/желтую/красную точку оценки;
- открыть график обычным сотрудником без доступа к ФОТ, плану выручки, ставкам и чужим деньгам;
- показывать сотрудникам минималистичные точки оценок в графике;
- использовать в ЛК визуальную модель старого графика: компактная сетка, цветные роли, колонка `Σ`, ФОТ/план и карточки итогов;
- ставить и снимать hourly-смены быстрым кликом по ячейке;
- выбирать фиксированную оплату смены из 3000/4000/6000/8000 или своей суммы.
- заводить и редактировать сотрудников, PIN, параметры графика и индивидуальные доступы к разделам через внутренний `/admin`.

Осталось для полного этапа:

- импорт реального JSON-бэкапа текущего графика;
- разрешить сотруднику уменьшать окончание своей смены.

## Этап 4. Закрытие смены

Исходный проект: `RestForm`.

Реализовано:

- раздел подключен как сервис `shift_close` по адресу `/smena`;
- доступ открыт только руководителю/менеджеру или сотрудникам с правом на сервис;
- форма пишет отчет в backend и Postgres без Google Sheets/Apps Script;
- backend считает кассу, безнал, общую выручку, выплату кальянщику, доп. расходы, ожидаемую кассу, отклонение и процент плана;
- `Нетмонет` добавлен в безнал;
- переводы участвуют в расчете остатка кассы;
- доп. расходы работают как динамический список `amount + comment`;
- фото чеков сохраняются по типам `terminal_1`, `terminal_2`, `shift_close`, `other`;
- Telegram-отчеты для руководителя и команды отправляются отдельно и не блокируют закрытие смены;
- ставка кальянщика хранится в карточке сотрудника.

Технические правила:

- смена, закрытая до `06:00`, относится к предыдущему рабочему дню;
- `opening_cash_expected` берется из `closing_cash_actual` предыдущей смены;
- `taxi_amount` не вычитается из кассы;
- `cash_diff_limit = 500`;
- `taxi_limit = 2000`;
- `revenue_plan` берется из модуля `График`;
- если Telegram недоступен, смена все равно закрывается;
- если PNG-отчет недоступен, используется текстовый fallback.

Осталось для развития:

- месячный дашборд поверх `shift_closings`;
- просмотр и выгрузка фото из интерфейса руководителя;
- полноценная генерация PNG-отчета вместо текущего текстового fallback.

## Этап 5. Задания и выплаты

Что сделать:

- раздел заданий;
- назначение задания сотруднику;
- дедлайн;
- статус выполнения;
- отображение в ЛК сотрудника;
- отображение выплат сотруднику;
- история фактических выплат;
- ближайшая плановая выплата.

Результат этапа:

- сотрудник видит свои задания;
- сотрудник видит часы, начислено, выдано, остаток;
- сотрудник видит ближайшую плановую выплату.

## Этап 6. Админка руководителя

Что сделать через NocoDB:

- сотрудники;
- PIN;
- доступы к сервисам;
- смены;
- выплаты;
- оценки;
- задания;
- отчеты закрытия смен.

Что нельзя делать через NocoDB:

- хранить открытые PIN;
- обходить бизнес-правила для сотрудника;
- отдавать NocoDB сотрудникам.

Результат этапа:

- руководитель может управлять данными через админку;
- сотрудник работает только через ЛК.

## Этап 7. Деплой

Что сделать:

- выбрать сервер;
- настроить домены;
- настроить HTTPS;
- настроить backup Postgres;
- настроить переменные окружения;
- настроить production Docker Compose;
- настроить обновление backend;
- закрыть NocoDB отдельным доступом.

Проверки перед запуском:

- вход по PIN работает;
- чужие данные не видны;
- backup базы создается;
- восстановление backup проверено;
- NocoDB недоступен сотрудникам;
- старый график можно оставить как резерв.

## MVP, с которого стоит начинать программирование

Минимальный первый релиз:

1. Backend + Postgres + NocoDB локально.
2. Таблицы сотрудников, PIN, сервисов и доступов.
3. Вход по PIN.
4. Главная страница ЛК с доступными кнопками.
5. Один сервис-заглушка `График`.
6. Импорт данных календаря из JSON.

После этого переносить полный календарь.
