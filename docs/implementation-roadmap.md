# План реализации ЛК Etna

## Принцип

Делаем сервис поэтапно. Сначала технический фундамент, потом переносим существующий календарь, затем подключаем закрытие смены и новые сервисы.

Текущий график на GitHub Pages остается рабочим до тех пор, пока новый ЛК не сможет заменить его без потери данных.

## Этап 0. Подготовка

Результат:

- зафиксирована архитектура;
- описана модель данных;
- выбрана связка `Postgres + backend + NocoDB`;
- определены домены;
- определены границы MVP.

Статус: текущий этап.

## Этап 1. Инфраструктура

Что сделать:

- создать новый backend-проект;
- добавить Docker Compose;
- поднять `Postgres`;
- поднять `NocoDB`;
- поднять backend API;
- настроить `.env.example`;
- добавить healthcheck API;
- добавить миграции базы;
- подготовить reverse proxy для доменов.

Минимальный состав контейнеров:

- `postgres`;
- `api`;
- `nocodb`;
- `caddy` или `nginx`.

Результат этапа:

- локально запускается база, API и NocoDB;
- можно открыть админку;
- можно выполнить первую миграцию;
- API отвечает `200` на `/health`.

## Этап 2. Авторизация и центральный ЛК

Что сделать:

- таблицы сотрудников;
- таблицы PIN-хешей;
- вход по PIN;
- защита от перебора PIN;
- сессии;
- список доступных сервисов;
- главный экран ЛК после входа.

MVP-экран:

- 4 точки PIN;
- после входа - кнопки сервисов;
- сотрудник видит только разрешенные сервисы.

Результат этапа:

- руководитель может завести сотрудника;
- сотрудник входит по PIN;
- backend возвращает список его сервисов.

## Этап 3. Перенос календаря

Что сделать:

- перенести модель сотрудников;
- перенести смены;
- перенести выплаты;
- перенести плановые дни зарплаты;
- перенести оценки;
- перенести дедлайны;
- сделать импорт JSON-бэкапа из текущего календаря;
- заменить `localStorage` на API.

Правила:

- руководитель видит полный график;
- сотрудник видит общий график, но персональные деньги только свои;
- сотрудник может уменьшить окончание своей смены;
- выплаты не меняют расчетный дневной ФОТ и план выручки.

Результат этапа:

- календарь работает внутри ЛК;
- данные хранятся в Postgres;
- старый JSON можно импортировать.

## Этап 4. Закрытие смены

Исходный проект: `RestForm`.

Что сделать:

- выделить из `RestForm` нужную форму;
- подключить ее как сервис `shift_close`;
- доступ открыть только выбранным сотрудникам;
- запись отчета отправлять в backend;
- backend пишет отчет в Postgres;
- при необходимости backend продолжает синхронизацию с Google Sheets и Telegram.

Рекомендуемый подход:

- не переносить весь тяжелый функционал сразу;
- сначала сохранить работающую форму закрытия смены;
- затем постепенно убрать лишнее и перевести критичные данные в Postgres.

Результат этапа:

- сотрудник входит в ЛК;
- видит кнопку `Закрытие смены`, только если ему разрешено;
- отправляет отчет;
- отчет появляется в Postgres и доступен руководителю.

## Этап 5. Задания и выплаты

Что сделать:

- раздел заданий;
- назначение задания сотруднику;
- дедлайн;
- статус выполнения;
- отображение в ЛК сотрудника;
- отображение выплат сотруднику;
- история фактических выплат;
- ближайшая плановая выплата.

Результат этапа:

- сотрудник видит свои задания;
- сотрудник видит часы, начислено, выдано, остаток;
- сотрудник видит ближайшую плановую выплату.

## Этап 6. Админка руководителя

Что сделать через NocoDB:

- сотрудники;
- PIN;
- доступы к сервисам;
- смены;
- выплаты;
- оценки;
- задания;
- отчеты закрытия смен.

Что нельзя делать через NocoDB:

- хранить открытые PIN;
- обходить бизнес-правила для сотрудника;
- отдавать NocoDB сотрудникам.

Результат этапа:

- руководитель может управлять данными через админку;
- сотрудник работает только через ЛК.

## Этап 7. Деплой

Что сделать:

- выбрать сервер;
- настроить домены;
- настроить HTTPS;
- настроить backup Postgres;
- настроить переменные окружения;
- настроить production Docker Compose;
- настроить обновление backend;
- закрыть NocoDB отдельным доступом.

Проверки перед запуском:

- вход по PIN работает;
- чужие данные не видны;
- backup базы создается;
- восстановление backup проверено;
- NocoDB недоступен сотрудникам;
- старый график можно оставить как резерв.

## MVP, с которого стоит начинать программирование

Минимальный первый релиз:

1. Backend + Postgres + NocoDB локально.
2. Таблицы сотрудников, PIN, сервисов и доступов.
3. Вход по PIN.
4. Главная страница ЛК с доступными кнопками.
5. Один сервис-заглушка `График`.
6. Импорт данных календаря из JSON.

После этого переносить полный календарь.
